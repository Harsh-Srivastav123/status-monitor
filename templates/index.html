How It Works Architecture

📡 RSS + ETag PRIMARY
Most efficient approach. Uses HTTP conditional GET with ETag headers. When content hasn't changed, server returns 304 Not Modified - zero bandwidth consumed. 
Every 60s:
  GET /history.atom
  Header: If-None-Match: "abc123"
     │
     ├── 304? → No change → Sleep
     │
     └── 200? → Parse new entries
               → Detect incidents
               → Print to console
🔄 Hash Polling FALLBACK
For providers without RSS feeds. Fetches JSON API endpoint and compares SHA256 hash of response body. Only processes when hash changes. 
Every 60s:
  GET /api/v2/incidents.json
     │
     └── SHA256(response)
            │
            ├── Same hash? → Sleep
            │
            └── Different? → Parse changes
                          → Detect incidents
                          → Print to console
🎯 Webhook IDEAL
Best approach when available. Provider pushes updates to our endpoint in real-time. Zero polling, instant detection, minimal resource usage. 
Provider Status Change:
     │
     └── POST /webhook/{token}
            │
            └── Validate HMAC signature
                  │
                  └── Parse payload
                        → Store incident
                        → Push to UI

Why This Is NOT Inefficient Polling

The assignment requirement states: "You should not rely on polling inefficiently — use a more event-based approach that can scale to 100+ status pages." Here's how this solution meets that requirement:

❌ Naive Polling (Bad) 
while True:
    response = requests.get(url)
    # Always downloads full page
    # Always parses everything
    # Wastes bandwidth & CPU
    parse_and_check(response)
    time.sleep(60)
# Blocking! Can only do 1 provider
Bandwidth: ~50KB/request
100 providers: 5MB/minute
Threading: 100 threads needed
✓ Our Approach (Efficient) 
async def monitor(provider):
    etag = None
    while True:
        headers = {"If-None-Match": etag}
        response = await session.get(url, headers)

        if response.status == 304:
            # No change! Zero bandwidth used
            pass
        else:
            etag = response.headers.get("ETag")
            await process_new_entries(response)

        await asyncio.sleep(60)
# Non-blocking! 100 providers in 1 thread
Bandwidth: ~0KB when unchanged
100 providers: ~50KB/minute*
Threading: Single thread (AsyncIO)

Event-Based Detection (Not Dumb Polling)

RSS + ETag = Conditional GET

HTTP standard mechanism. Server tracks content version via ETag header. Client sends If-None-Match header. If content unchanged, server returns 304 Not Modified with empty body. This is the web's built-in "tell me only if something changed" mechanism.

Request:
  GET /history.atom HTTP/1.1
  If-None-Match: "abc123"

Response (no change):
  HTTP/1.1 304 Not Modified
  (empty body - zero bandwidth!)

Response (changed):
  HTTP/1.1 200 OK
  ETag: "def456"
  (new content to parse)
Hash Polling = Change Detection

For APIs without ETag support, we compute SHA256 hash of response body and compare with cached hash. Only parse when hash differs. Minimizes CPU usage - hash comparison is O(1).

# First request
hash1 = sha256(response.body)  # "a1b2c3..."
cache[provider] = hash1

# Next request
hash2 = sha256(response.body)  # "a1b2c3..."
if hash2 == cache[provider]:
    pass  # Skip parsing! No change
else:
    parse_incidents(response)
    cache[provider] = hash2

Production Upgrades Needed

1. Add Message Queue (Kafka)

Decouple event detection from processing. Never lose incidents even if downstream fails.

Topics:
  - status.raw_events    (all monitors publish here)
  - status.incidents     (normalized, deduplicated)
  - status.updates       (incident updates)

2. Add Redis Cache

Fast state storage for ETags, hashes, rate limiting. No SQL overhead for hot data.

Keys:
  etag:{provider_id}          → "abc123"
  last_hash:{provider_id}     → "sha256:..."
  rate_limit:{provider_id}    → request_count (TTL: 60s)

3. Add PostgreSQL

Persistent storage for providers, incidents, and history.

Tables:
  providers        → config, mode, URLs
  incidents        → detected incidents
  incident_updates → timeline of changes

4. Add Webhook Support

Generate unique webhook URLs for providers that support push notifications.

POST /api/providers/{id}/webhook/generate
  → Returns: /webhook/{uuid_token}
  → Provider POSTs updates here
  → Validate HMAC signature
  → Instant incident detection

5. Real-time WebSocket

Push incidents to UI instantly instead of polling with auto-refresh.

WS /ws/stream
  → Kafka consumer pushes events
  → UI updates in real-time
  → No page refresh needed
